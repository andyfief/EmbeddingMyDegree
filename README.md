# School Knowledge Base — Semantic Search Pipeline

A personal RAG (Retrieval-Augmented Generation) pipeline that ingests an entire school directory, OCRs image-based PDFs locally, embeds everything with OpenAI, and enables semantic search over the full corpus.

---

## Fast Facts

| | |
|---|---|
| **Embedding model** | `text-embedding-3-small` (1536 dimensions) |
| **Embedding storage** | AWS S3 (JSONL format) |
| **Query log storage** | AWS DynamoDB |
| **OCR engine** | `marker-pdf` (GPU-accelerated) |
| **OCR hardware** | NVIDIA RTX 5060 Ti — full corpus OCR in ~5 hours |
| **Chunk size** | 6,000 chars (~1,500 tokens) with 400-char overlap |
| **Embedding cost** | ~$0.02 / 1M tokens (standard) or $0.01 / 1M (Batch API) |

## Scripts

### `fileEDA.py` — Directory Scanner
Walks the `../../School` directory and counts files by extension. Skips junk directories (`node_modules`, `.git`, `venv`, etc.) and noisy paths (`site-packages`, `huggingface`, etc.). Outputs `school_scan_results.json`.

### `ingest.py` — Chunker
Reads `school_scan_results.json`, re-walks the directory, and processes every file type:
- **Plaintext / code / JSON / notebooks** — character-based chunking
- **PDF** — `pymupdf` text extraction; pages with fewer than 20 non-empty lines are flagged for OCR
- **DOCX** — `python-docx`
- **PPTX** — `python-pptx`, with per-slide markers
- **XLSX** — `openpyxl`, header row repeated per chunk
- **Images** — stub record (no text)

All chunks share a consistent format: `path`, `header` (first 1,000 chars), and `body`.

### `ocr_marker.py` — GPU OCR
Reads `ocr_candidates.txt` and `ingest_errors.log` to find image-based PDF pages skipped during ingestion. Runs `marker-pdf` page-by-page with `force_ocr=True` using a local CUDA GPU. Supports resume — already-processed files are skipped. Outputs `marker_chunks.jsonl`.

### `embed_chunks.py` — Embedder
Reads a JSONL of chunks (typically `marker_chunks.jsonl` which combines both sources), estimates token count and cost, then batches requests to OpenAI's `text-embedding-3-small` in groups of 20. Handles oversized chunks (>8,000 tokens) by splitting with overlap. Supports resume via chunk ID tracking. Outputs `chunks_embedded.jsonl`.

### `search_chunks.py` — Semantic Search
Loads `chunks_embedded.jsonl` into a normalized NumPy matrix, embeds a query string, and returns the top-K results by cosine similarity.

### `review_chunks.py` — QA Browser
Interactive CLI for auditing chunk quality. Supports filtering by extension, keyword search, stats-only mode, and full export.

## Chunk Format

```json
{
  "file_path": "cs_461/lecture_notes.pdf",
  "ext": ".pdf",
  "category": "document",
  "chunk_index": 3,
  "total_chunks": 12,
  "start_char": 16600,
  "end_char": 22600,
  "start_page": 4,
  "text": "path: cs_461/lecture_notes.pdf\nheader:\n...\nbody:\n...",
  "embedding": [0.0012, -0.0431, ...]   // 1536 floats, present after embed step
}
```

OCR chunks carry two additional fields: `"ocr": true` and `"ocr_engine": "marker"`.

## Search Application

A full-stack semantic search UI built on top of the pipeline. The backend loads the embedded JSONL from S3 at startup, scores queries via cosine similarity, and logs every search to DynamoDB. The frontend is a dark-themed React/TypeScript app served via Vite.

### App Structure

```
search-app/
├── backend/
│   ├── main.py              # FastAPI app — /search and /open endpoints
│   ├── s3_loader.py         # Downloads chunks_embedded.jsonl from S3 into memory
│   ├── searcher.py          # NumPy cosine similarity search
│   ├── dynamo_logger.py     # Logs query, top result, latency to DynamoDB
│   └── requirements.txt
└── frontend/
    ├── index.html
    ├── package.json
    ├── vite.config.ts
    ├── tsconfig.json
    └── src/
        ├── main.tsx
        ├── App.tsx
        ├── api.ts           # fetch wrappers for /search and /open
        ├── types.ts         # ChunkResult, SearchResponse interfaces
        ├── index.css        # Dark theme
        └── components/
            ├── SearchBar.tsx
            └── ResultCard.tsx
```

### How It Works

1. **Startup** — `s3_loader.py` pulls `chunks_embedded.jsonl` from S3 and builds a normalized NumPy matrix in memory (one-time load)
2. **Search** — query text is embedded via OpenAI, dot-producted against the matrix, top-K results returned with score, preview, file path, and chunk metadata
3. **Open** — `ResultCard` has an "Open" button that calls `/open`, which runs `os.startfile()` (Windows) to open the source file locally
4. **Logging** — every query, result count, top score, and latency is written to DynamoDB for usage tracking

### AWS Resources

| Resource | Purpose |
|---|---|
| S3 bucket | Stores `chunks_embedded.jsonl` |
| DynamoDB table (`search_queries`) | Query log — timestamp, query text, top result, latency |
| IAM user | Scoped to `s3:GetObject` on the bucket + `dynamodb:PutItem` on the table |
